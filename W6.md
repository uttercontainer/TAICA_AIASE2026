## Part 1：Harness Design for Long-Running Apps
### 核心問題：為什麼單一 Agent 無法獨撐？

兩個反覆出現的結構性困境：

- **Context Anxiety（情境焦慮）**
  - Context Window 接近上限時，模型開始「搶著收尾」，品質急劇下滑
  - Compaction 無法根治——真正需要的是完整的 **Context Reset** 機制

- **Self-Evaluation Failure（自我評估失靈）**
  - 請模型評估自己的輸出，幾乎永遠打高分
  - 主觀任務（如 UI 設計）尤為嚴重，客觀任務（如功能測試）同樣存在判斷偏差

> **本質**：「單一 Agent 同時負責執行與評估」本身就是結構性缺陷。

---

### 靈感來自 GAN 架構

**Generative Adversarial Networks 核心精神**
- 生成器（Generator）←→ 判別器（Discriminator）
- 兩個角色的對抗，驅動彼此不斷進步

**移植到 Agent 設計**
- 把「做事的人」和「挑毛病的人」角色分離
- 調教一個專門挑剔的 Evaluator，比讓 Generator 自我批判容易太多
- 外部回饋一旦存在，Generator 就有具體的目標可以迭代

> *"Tuning a standalone evaluator to be skeptical is far more tractable than making a generator critical of its own work."*

**延伸**：`openai/codex-plugin-cc` — 從 Claude Code 呼叫 Codex 互審

### 先從 Frontend Design 開始驗證

Claude 的 UI 輸出太「安全」、缺乏設計個性 → 解方：把「美感」量化為可評分標準

| 評分維度 | 說明 | 重要性 |
|---------|------|--------|
| Design Quality | 整體一致的視覺語言與個性 | ⭐⭐⭐⭐⭐ |
| Originality | 有真正的設計決策？非 AI Slop | ⭐⭐⭐⭐⭐ |
| Craft | 間距、對比、字體層級等技術執行 | ⭐⭐ |
| Functionality | 使用者能否完成核心任務 | ⭐⭐⭐ |

**明確懲罰 AI 慣用模式**
- 紫色漸層白底卡片 → 直接扣分
- 未修改的 Stock Components → 直接扣分

**流程**：Generator → Playwright Evaluator → 5～15 輪迭代 → 突破性設計出現

### Three-Agent Architecture for Frontend Coding

```
User Prompt → Planner → Generator（逐 sprint）→ Evaluator（Playwright）
```

| 角色 | 類比 | 職責 |
|------|------|------|
| 🧭 Planner | PM | 模糊需求 → 完整規格（16 features / 10 sprints） |
| 🛠 Generator | 工程師 | React + Vite + FastAPI + SQLite 逐 sprint 實作 |
| 🔍 Evaluator | QA | Playwright 實際操作，找真實 Bug |

---

### Sprint Contract：對齊的關鍵機制

**每個 Sprint 開始前的三步驟**
1. **Generator 提出**：這個 Sprint 我要做什麼、如何驗收——先說清楚再動手
2. **Evaluator 審核**：確認測試標準合理、範圍正確、邊界條件覆蓋完整
3. **雙方同意後才寫 Code**：先定義「完成」的標準，大幅減少來回修改的浪費

**為什麼 Sprint Contract 如此重要？**
- Product Spec 故意保持 High-Level，避免 Planner 的錯誤向下傳播
- Sprint Contract 填補 User Story → 可測試實作 之間的 Gap
- 本質上就是軟體工程中的 Sprint Planning，只是執行者換成了 AI Agent

### 數據說話：單一 Agent vs. 三 Agent Harness

**案例一：2D Retro Game Maker**

| 指標 | Solo Agent | Full Harness |
|------|-----------|--------------|
| 時間 | 20 分鐘 | 6 小時 |
| 成本 | $9 | $200 |
| 結果 | 核心功能壞掉，無法遊玩 | 完整可運作，有物理引擎、AI 關卡生成器 |

**Evaluator 抓到的真實 Bug（Solo 完全看不見）**
- `PUT /frames/reorder` 路由定義在 `/{frame_id}` 之後，FastAPI 把字串 `"reorder"` 當成 integer 解析，回傳 422 錯誤

> **結論**：20 倍的成本換來的是「能用」與「不能用」之間的本質差異，而非邊際改善。

### Harness 的演進：三個階段

| Stage | 模型 | 架構 | 重點 |
|-------|------|------|------|
| Stage 1 | Sonnet 4.5 (2025 Nov) | 2-Agent：Initializer + Coding Agent | Context Reset 是核心突破；發現 Self-Evaluation 問題 |
| Stage 2 | Opus 4.5 | 3-Agent：Planner + Generator + Evaluator | 引入 Sprint Contract 與 GAN-inspired feedback loop；首次達到「真正能用」標準 |
| Stage 3 | Opus 4.6（簡化版） | 移除 Sprint 結構 | Evaluator 改為單次 End-of-Run 評估；效果維持，複雜度大幅降低 |

---

### 最重要的原則：Harness 應隨模型進化而簡化

> *"Every component in a harness encodes an assumption about what the model can't do on its own."*

**具體實例**
- Opus 4.5 需要 Sprint 分解 → Opus 4.6 原生支援長時間連貫作業 → Sprint 結構可拆掉
- Sonnet 4.5 的 Context Anxiety 嚴重 → 需要 Context Reset → 新模型已不需要

**實踐建議**
1. 每次新模型發布，對 Harness 做一次「稽核」
2. 問自己：「這個元件是因為模型的哪個限制而存在的？那個限制現在還成立嗎？」
3. 移除不再 Load-bearing 的支架——技術債要定期清理

> 延伸思考：團隊流程、管理制度，是否也有「以前出過事所以加上去」、卻早已不再必要的機制？

---

### Multi-Agent 開發核心精神

**架構層面**
- 角色分離是突破 AI 品質天花板的關鍵槓桿
- Sprint Contract = 動手前先對齊「完成」的定義
- File-based Communication 讓 Agent 間的交接乾淨、可追蹤

**成本層面**
- 20x 成本換來的是「能用」vs「不能用」的本質差異
- Evaluator 不是固定成本——任務越在模型能力邊界，Evaluator 越值錢

**迭代層面**
- 從最簡單的 Harness 開始，只在需要時增加複雜度
- 模型進步 → Harness 應該變簡單，不是更複雜
- 評估標準（Criteria）本身就是一種 Prompt Engineering

**管理類比：人類團隊智慧 × AI Harness 設計**

| 人類團隊智慧 | AI Harness 設計 |
|------------|----------------|
| 工程師不該自己做 QA | Generator ≠ Evaluator |
| 需求不清楚就開始做 = 浪費 | Planner 先出 Spec |
| PM 定義驗收標準 | Sprint Contract |
| 不需要的流程要定期檢視 | Harness 隨模型進化縮減 |

## Part 2：軟體開發與維護是一場等價交換的遊戲

### 核心命題：In the Past, It is True

> 「你在前期省下的，後期一定會加倍付出。」
> 這不是詛咒，而是軟體系統的物理定律。

**AI 時代的新現實**
- AI 讓「前期省略」變得更容易、更誘人
- 技術債的積累更隱蔽、爆發更劇烈
- 核心命題不但沒有失效，**反而更加成立**

| | 原版的等價交換 | AI 時代的新現實 |
|--|--------------|----------------|
| 等式 | 前期投入 ←→ 後期擴展能力 | 前期投入（速度快 × AI）←→ 後期擴展能力（？） |
| 特點 | 時間與心力的付出換取穩定、可維護的系統 | 技術債被 AI 掩蓋了，問題更難被早期發現 |

---

### 等價交換的本質

**AI 改變的不是等價交換的本質，而是「交換的速率」**

| 維度 | 傳統開發 | AI 輔助開發 |
|------|---------|-----------|
| 前期快速路徑的速度 | 快 | 快 × 10 |
| 技術債的累積速率 | 慢慢堆 | 快速堆疊 |
| 技術債的可見度 | 明顯 | 更隱蔽（AI 讓 code 看起來整齊） |
| 後期爆炸的規模 | 可管理 | 更難收拾 |

**AI 時代的新等式**
- Prompt 品質 ↔ AI 產出的可維護性
- Spec 精確度 ↔ Agent 執行的可預測性
- Harness 設計 ↔ 自動化系統的長期穩定性

---

### 前期投入高 → 後期穩定、可擴展

原版清單依然全面有效；AI 時代在此基礎上增添全新前期投入維度：

| AI 時代前期投入 | 對應原版概念 |
|--------------|------------|
| **Spec-Driven Development** | 完善需求分析 |
| **AGENTS.md / CLAUDE.md** | 架構設計文件 |
| **Harness Engineering**（Planner / Generator / Evaluator） | CI/CD 與測試策略 |
| **Eval-Driven Development** | 自動化測試 |
| **Context Engineering** | 可觀察性部署 |

> **核心原則**：Harness 的設計本質上是在「編碼模型的限制假設」。前期投入建立好 Harness，就是前期投入建立好架構的 AI 時代版本。

---

### 前期輕量化 → 後期風險

**五種 AI 時代新型陷阱**

- ❌ **讓 AI 一次寫完所有 code，不做 review**
  - AI 有 Inflated Self-Assessment，無法客觀識別自身產出的問題
- ❌ **沒有 Spec 就叫 Agent 開始做**
  - Agent 會 under-scope；沒有規格就無從驗收
- ❌ **把 AI 產出的 code 當「能跑就好」**
  - 功能表面正常，但核心邏輯可能是壞的
- ❌ **Harness 設計完之後從不重新評估**
  - 模型進步了，舊的 Scaffolding 反而成為技術債
  - Ref：「模型會把你的 scaffolding 當早餐吃掉」
- ❌ **用 AI 快速堆疊功能，跳過測試**
  - AI 生成的 code 看起來整齊，但測試覆蓋率為零

> **新版等式**：Vibe Coding 的速度 × AI 的產能 = 技術債的累積速率

---

### 技術債在 AI 時代的新面貌

**五種更隱蔽、更危險的技術債類型**

| 技術債類型 | 描述 |
|---------|------|
| **Context 污染債** | Agent 後段開始「焦慮」，品質下滑；缺乏 Context Reset 機制，整個 Run 前功盡棄 |
| **Self-Eval 幻覺債** | AI 說「我做好了」但功能根本壞掉；沒有外部 Evaluator，問題沉默累積 |
| **Prompt 腐爛債** | 模型更新後，舊 Prompt 行為改變；沒有版本管理，結果難以重現 |
| **Harness 過時債** | 模型進步了，舊 Scaffolding 還在；不必要的複雜度拖慢整個系統 |
| **Spec 缺失債** | AI 做了很多，但做的不是你要的；沒有 Spec 的 AI 輸出根本無法驗收 |

> 舊規則在 AI 時代的新形式：「想重構時發現下不去手」→「想換模型、換框架時發現全部都要重做，因為早期沒有把 AI 行為抽象化」

### 實戰數據：等價交換的 AI 時代版本

**2D Retro Game Maker 案例**

| 比較維度 | 單一 Agent（快速路徑） | 三 Agent Harness（前期投入） |
|---------|-------------------|--------------------------|
| 時間 | 20 分鐘 | 6 小時 |
| 成本 | $9 | $200（約 22 倍） |
| 結果 | 功能壞掉，Game 無法遊玩 | 完整可運作，有物理引擎 |
| 技術債 | 高（需從頭重寫） | 低（有 Spec、有測試記錄） |

**Evaluator 抓到的真實 Bug**
- **路由錯誤**：FastAPI 路由順序錯誤 → 422 Error，API 完全無法正常響應
- **事件未觸發**：矩形填充工具 mouseUp 事件未正確觸發，核心繪圖功能失效
- **邏輯判斷錯誤**：Entity 刪除邏輯條件判斷錯誤，資料狀態管理完全混亂

> 你省下的那 5 小時 40 分鐘和 $191，換來的是一個「看起來有做完」但核心功能壞掉的產品。這就是 AI 時代的等價交換。

### Cloud DB vs Self-hosted DB：AI 時代新維度

| 考量維度 | Cloud DB | Self-hosted DB | AI 時代新增考量 |
|---------|---------|----------------|--------------|
| 前期成本 | 低 | 高 | AI 可幫你搭建，降低前期成本 |
| 規模成本 | 費用暴增 | 可控 | AI 可優化 Query，但治標不治本 |
| 擴展彈性 | 受限 | 高 | 與 AI Agent 整合的 Schema 設計更重要 |
| Vendor lock-in | 高 | 無 | AI 框架也有 lock-in（API 依賴） |

**LLM API 依賴的抉擇**
- 選擇一：快速接入 Managed AI API → 前期快，後期面臨 API 漲價、模型棄用、行為改變
- 選擇二：前期建立 LLM Router（如 LiteLLM / AWS Bedrock）→ 前期多花時間，後期可自由切換

### AI 時代的等價交換總表（四象限）

| | 前期投入低 | 前期投入高 |
|--|----------|----------|
| **AI 加速高** | 🔴【最危險】AI 快速堆疊，無品管；技術債 × AI 速度；看起來做完，其實全壞；無法驗收的 AI 垃圾場 | 🟢【最理想】SDD + Harness Engineering；AI 助力 × 品管到位；Planner → Generator → Evaluator；可驗收、可迭代、可維護 |
| **AI 加速低** | ⬛【傳統慢速】手工 Vibe Coding；老問題，慢慢爆；技術債累積速度較慢但仍會爆 | 🔵【傳統穩健】傳統架構設計；完善需求 + 測試 + 監控；慢但穩，後期可維護 |

> **AI 時代最危險的象限**：有 AI 加速 + 沒有 Harness 品管 → 用 AI 的速度，快速累積無法驗收的技術債。這個象限在傳統開發中根本不存在，是 AI 時代獨有的新風險。

### AI 時代的 Harness = 前期投入的新形式

| 角色 | 等價於 | 說明 |
|------|--------|------|
| **Planner**（PM 角色） | 完善需求分析 | 把模糊 prompt 展開成完整 Product Spec |
| **Generator**（工程師角色） | 模組化開發 | Sprint-by-Sprint，前期設計 Sprint Contract |
| **Evaluator**（QA 角色） | 自動化測試 + CI/CD | 用 Playwright 實際操作，前期調教嚴格標準 |

> **最關鍵的洞見**：調教一個獨立的 Evaluator 來挑剔，遠比讓 Generator 自我批判容易太多。這也是為什麼公司需要 QA 部門，而不是讓工程師說「我測過了沒問題」。

### AI 時代的 Value-Added Knee Point

傳統職場成長曲線是線性的（新人 → 中堅 → 專家），AI 時代的成長曲線出現明顯的關鍵轉折點。

定義不是「會用 AI 工具」，而是「能設計讓 AI 產出可驗收的系統」。

| 角色 | 能力描述 | 位置 |
|------|---------|------|
| **Vibe Coder** | 會下 Prompt，讓 AI 生成 code | 零價值區，可被 AI 直接取代 |
| **Harness Designer** | 能設計 Evaluator、寫 Sprint Contract | 跨越關鍵轉折，比 AI 單打獨鬥更強 |
| **Agentic Engineer** | 能架構 Planner-Generator-Evaluator 系統 | AI × 人類架構思維的最大化，高價值爆發區 |

### 總結

> 你在前期多花的每一分時間，都是在為後期的可持續維護與規模擴張蓄能。
> AI 讓前期的「省略」快了十倍，讓技術債的積累也快了十倍。**等價交換的物理定律沒有改變 — 只是交換的速率提高了。**

**三大核心原則**

| # | 原則 | 說明 |
|---|------|------|
| ① | **Spec First** | 沒有 Spec 的 AI 輸出，沒有辦法驗收。規格是一切的起點。 |
| ② | **Always Verifiable** | AI 說「做完了」不算，Evaluator 說才算。外部驗收是品質的唯一保證。 |
| ③ | **Reality Drives Iter** | Harness 要跟著模型能力演進，不是一次設計永久不變。主動縮減才是還清技術債。 |

> 軟體工程不是只看現在跑不跑得動，而是看六個月後你是否還能承受維護成本。**在 AI 時代，這個時間窗口縮短了 — 六週就可能讓你的 codebase 變成無法接手的 AI 垃圾場。**

### 最新使用趨勢：兩條分叉的曲線

## Part 4：Retrieval-Augmented Generation（RAG）

### 核心文獻

- Gao et al., *RAG for LLMs: A Survey*（arXiv 2312.10997, 2024）— 三範式分類框架的奠基之作
- Fan et al., *A Survey on RAG Meeting LLMs*（arXiv 2405.06211, KDD 2024）— RA-LLMs 架構、訓練、應用全覽
- Li et al., *A Survey of RAG-Reasoning Systems*（EMNLP Findings 2025）— RAG × Reasoning 最新前沿
- Awesome Generative AI Guide RAG Table（aishwaryanr/awesome-generative-ai-guide, 2023–2025）

---

### 為什麼需要 RAG？— LLM 的三大內生限制

| 限制 | 說明 |
|------|------|
| **Hallucination（幻覺）** | 模型生成看似合理但事實錯誤的內容；法律查詢幻覺率高達 69–88%（Dahl et al., 2024） |
| **Knowledge Cutoff（知識截止）** | 訓練資料有截止日期，無法掌握最新資訊；知識呈靜態凍結狀態 |
| **Non-transparency（不透明性）** | 推理過程難以追溯，缺乏可解釋性；用戶無法驗證答案來源 |

**三種解決方案比較**
- **Fine-tuning**：成本高、需大量標注、知識更新困難
- **Prompt Engineering**：靈活但受 context window 限制
- **RAG ✓**：平衡成本與效果，支援動態知識更新

> 💡 **直覺類比**：RAG 就像讓 LLM 在**開卷考試**作答，而非純粹靠記憶。答案有文件佐證，可溯源驗證，知識實時更新，只需更新外部資料庫而無需重新訓練模型。

---

### RAG 全局觀：Naive RAG 基礎（三步驟）

**流程**：索引（Indexing）→ 檢索（Retrieval）→ 生成（Generation）

- **Indexing**：將文件切割為 chunks 並向量化儲存
- **Retrieval**：將查詢向量化後計算語意相似度取回 Top-k
- **Generation**：將問題與 chunks 拼接為 prompt 送入 LLM

**Naive RAG 的五大缺陷**
- **Retrieval 精度不足**：檢索到語意相似但事實無關的文件，引入雜訊
- **Context Window 限制**：Top-k chunks 可能超過模型可處理長度
- **冗餘資訊**：多份 chunks 包含重複內容，稀釋有效信息密度
- **多跳推理失敗**：需要跨多份文件推理時表現差，單步檢索不足
- **生成與檢索脫節**：LLM 可能忽略檢索文件，仍靠參數記憶作答

---

### Advanced RAG

**Pre-Retrieval 技術**
- **Query Expansion**：擴充查詢詞彙，提高召回率
- **Query Rewriting**：LLM 輔助改寫，使查詢更適合語意檢索
- **HyDE**：先生成假設性答案，再用答案去檢索相關文件
- **Multi-query**：從多角度生成子查詢並聚合結果

**Post-Retrieval 技術**
- **Re-ranking**：用 Cross-encoder 或 Cohere Rerank 重新排序 Top-k 結果
- **Context Compression**：壓縮 chunks，僅保留與查詢最相關的句子（LLMLingua 等）
- **Filtering**：過濾低相關性 chunks，避免雜訊干擾 LLM 生成

> **HyDE（Hypothetical Document Embedding）**：在查詢時先讓 LLM 生成一份「假設性答案文件」，再以這份文件的 embedding 進行檢索，比原始查詢 embedding 更能捕捉文件空間的語意分佈。

---

### Modular RAG 模組化（最新一代架構）

將整個 pipeline 分解為可獨立優化、自由組合的功能模組，支援任意拓撲的組合、並行與替換。

| 模組 | 功能 |
|------|------|
| **Search Module** | 多來源異質資料檢索，支援 BM25 + Dense + Graph 混合搜尋 |
| **Memory Module** | 管理短期與長期記憶，包含對話歷史、用戶偏好 cache |
| **Fusion Module** | 聚合多路檢索結果，採用 Reciprocal Rank Fusion（RRF）等算法 |
| **Route Module** | 動態路由查詢至最佳資料源，透過分類器或 LLM 判斷 |
| **Predict Module** | 決定是否需要進一步檢索（如 Self-RAG 的 critique token 機制） |

**三代 RAG 架構比較**

| 維度 | Naive RAG | Advanced RAG | Modular RAG |
|------|----------|--------------|-------------|
| 架構複雜度 | 線性 | 擴充 pre/post | 任意組合圖 |
| 靈活性 | 低 | 中 | 高 |
| 多跳推理 | 差 | 中等 | 良好 |
| 生產部署難度 | 低 | 中 | 高 |
| 代表系統 | 基本 QA bot | FLARE, RAG-Fusion | Self-RAG, MRAG |

---

### RAG 的三大技術支柱：Retriever

**Sparse Retrieval（BM25）**
- 基於精確詞彙匹配，計算詞彙頻率與文件長度懲罰
- 速度快、可解釋、無需訓練
- 無法處理語意相似而用詞不同的查詢

**Dense Retrieval（DPR / BGE / E5）**
- 雙 Encoder 架構，Query encoder + Passage encoder 分別映射至向量空間
- 以 cosine similarity 計算相關度
- BGE / E5 為目前多語言 dense retriever 的 SOTA

**Hybrid Retrieval（混合策略）**
```
Hybrid = α×BM25 + (1-α)×Dense
RRF(d) = Σ 1/(k + rank_i(d))  (k=60)
```

**Advanced Retrieval 策略**
- **ColBERT（Late Interaction）**：Query 與 Passage tokens 逐一比較，細粒度匹配
- **SPLADE**：稀疏向量，兼具可解釋性與語意能力
- **HyPE**：Indexing 階段預先生成假設問題，提升 42% precision

---

### Augmentation 增強技術

**Chunking 策略**
- **固定大小分割**：512 tokens，簡單但可能截斷語意
- **語意分割**：偵測段落邊界，保留完整語意單元
- **Sliding Window**：相鄰 chunk 有重疊，減少邊界資訊遺失
- **Hierarchical**：多層索引（段落→句子→Token），支援不同粒度查詢

**文件增強技術**
- **Question Generation**：對每份 chunk 生成可能問題，建立 question-document 映射
- **Summary Indexing**：長文件先摘要再索引，保留高層次語意
- **Metadata Enrichment**：附加標題、作者、日期等結構化 metadata

**Lost in the Middle 問題（Liu et al., 2024）**
LLM 對 context 開頭與結尾的注意力顯著高於中間部分，置於中段的關鍵資訊容易被忽略。

**Context 管理解決策略**
- Re-ranking：將最相關 chunk 置於 prompt 首尾位置
- Context Compression：用 LLM 壓縮 chunk，只保留關鍵句（如 LLMLingua）
- Selective Context：根據 token 重要性分數動態過濾 context 內容

> **Context Engineering** 是 RAG 系統中最容易被忽視但影響最大的環節之一。

---

### Generator 生成器設計

**標準 RAG Prompt 結構**
```
System: 你是客服助理，只依據以下文件回答問題，若文件中找不到相關資訊請說「目前我無法回答此問題」

Context:
[Document 1]: {chunk_1}
[Document 2]: {chunk_2}
[Document 3]: {chunk_3}

Question: {user_query}
Answer:
```

**關鍵設計原則**
- 明確角色定義（限制 LLM 只依賴 context）
- 引用格式要求（標注文件編號）
- 不確定性表達（允許「資訊不足」回應）

**生成策略分類**
- **Single-pass**：一次性生成最終答案，適合簡單問答
- **Iterative Refinement**：逐塊送入 context，迭代精化答案，適合長文件
- **Query Decomposition**：先分解複雜問題，再逐一回答子問題，適合多跳推理
- **Chain-of-Thought**：要求 LLM 展示完整推理過程，提升可解釋性

---

### Advanced RAG 技術深探

**Self-RAG（Asai et al., 2023）**
引入四種特殊 reflection token 讓 LLM 自主控制檢索行為：
- `[Retrieve]`：決定是否觸發檢索
- `[IsREL]`：判斷段落相關性
- `[IsSUP]`：判斷生成是否有文件支撐
- `[IsUSE]`：評估答案有用性

**FLARE（Active Retrieval）**
Forward-Looking Active REtrieval：LLM 生成下一句時，若出現低機率 token（不確定），則觸發檢索。「generate-then-retrieve」模式比傳統「retrieve-then-generate」更靈活。

**GraphRAG（Edge et al., 2025）**
文件 → 抽取知識三元組 → 建構 Knowledge Graph → 社群偵測（Community Detection）→ 多層次摘要 → 查詢時聚合社群摘要。擅長「整個語料的主題與趨勢分析」等全局性問題。

**HippoRAG（Gutiérrez et al., 2025）**
模擬海馬體記憶機制，建構概念／事實／情節三層記憶結構，以 Personalized PageRank 在知識圖上尋路，特別適合複雜多跳推理。

**RAG-Fusion**
從查詢生成 N 個重述版本分別檢索，透過 RRF 融合排序，從多角度覆蓋查詢空間，顯著提升召回率。

---

### Agentic RAG 與 RAG-Reasoning 系統（Li et al., EMNLP 2025）

**三大系統架構**

1. **Reasoning-Enhanced RAG**：用 Reasoning 優化 RAG 每個階段
   - CoT 分解複雜查詢
   - Chain-of-Note：先對文件寫筆記再整合
   - TRACE：建構知識導向推理鏈

2. **RAG-Enhanced Reasoning**：用 RAG 補充 Reasoning 所需知識前提
   - KG-RAG：從知識圖譜提取多跳推理路徑
   - Think-on-Graph：在知識圖上進行 beam search 逐步找到推理路徑

3. **Synergized RAG-Reasoning**：Reasoning 與 Retrieval 交替迭代相互增強
   - Search-R1：用 RL 學習何時搜尋
   - DeepResearcher：端到端 RL + 真實網頁瀏覽
   - MCTS-RAG：樹狀搜尋空間探索

**Agentic RAG 架構**
當 RAG 與 Agent 結合，Retrieval 不再是靜態 pipeline，而是 Agent 的動態工具。Agent 可動態選擇 Web Search、Vector DB Search、Knowledge Graph Query、Calculator 等工具並迭代組合。

**Multi-Agent RAG 架構**
- **集中式（Centralized）**：主 Agent 協調多個 Retrieval 子 Agent（代表：HM-RAG、Chain of Agents）
- **分散式（Decentralized）**：多 Agent 各自獨立運作並協商（代表：MRAG、MDocAgent）

**代表系統一覽**

| 系統 | 方法 | 特點 |
|------|------|------|
| Search-R1 | RL 訓練 + 搜尋引擎整合 | 用 RL 學習何時搜尋、搜尋什麼 |
| Search-o1 | 思考鏈 + 主動搜尋 | 結合 OpenAI o1 式長鏈推理 |
| DeepResearcher | 端到端 RL + 真實網頁瀏覽 | 最接近真實研究行為的系統 |
| MCTS-RAG | Monte Carlo Tree Search + RAG | 樹狀搜尋空間系統性探索 |
| ReaRAG | 迭代檢索 + 增強推理 | 優化大型推理模型的事實性 |

### RAG 評估框架

**三大評估維度**

| 維度 | 指標 |
|------|------|
| **Retrieval Quality** | Context Precision（Top-k 中真正相關的比例）、Context Recall（取回完整度）、Context Relevance |
| **Generation Quality** | Answer Faithfulness（忠實於 context，幻覺檢測）、Answer Relevance、Groundedness |
| **End-to-End Quality** | EM（Exact Match）、F1 Score（詞彙重疊）、ROUGE / BLEU |

**RAGAs 框架（Es et al., EACL 2024）**
目前最廣泛使用的 RAG 自動評估框架，無需人工標注，以 LLM 作為評判者自動計算四大核心指標：
```python
from ragas import evaluate
from ragas.metrics import (
    faithfulness, answer_relevancy,
    context_precision, context_recall
)
results = evaluate(dataset=your_dataset, metrics=[...])
```

**常見評估陷阱**
- 只評估 End-to-End：無法診斷問題來自 Retrieval 還是 Generation
- 忽略 Context Precision：大量無關 context 拉低生成品質，卻常被整體指標掩蓋
- 用 EM 評估開放式問題：答案的多樣性使 EM 嚴重低估實際表現
- 數據污染：Benchmark 可能出現在 LLM 訓練資料中，導致虛高分數

### 從 Naive 到 Production：LangChain + ChromaDB 最小可行 RAG

**Naive RAG Pipeline**
```python
from langchain.document_loaders import DirectoryLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.embeddings import OpenAIEmbeddings
from langchain.vectorstores import Chroma
from langchain.chat_models import ChatOpenAI
from langchain.chains import RetrievalQA

# Step 1: Indexing
loader = DirectoryLoader("./docs/", glob="**/*.pdf")
documents = loader.load()
splitter = RecursiveCharacterTextSplitter(chunk_size=512, chunk_overlap=50)
chunks = splitter.split_documents(documents)

# Step 2: Embedding & Vector Store
embeddings = OpenAIEmbeddings()
vectorstore = Chroma.from_documents(chunks, embeddings)

# Step 3: Retrieval + Generation
llm = ChatOpenAI(model="gpt-4o", temperature=0)
qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    retriever=vectorstore.as_retriever(search_kwargs={"k": 5})
)
answer = qa_chain.invoke("你的問題")
```

**Production RAG 設計清單**

- **Indexing 層**
  - 根據文件類型選擇 chunking 策略（PDF/HTML/Code 各異）
  - chunk_size 256–1024 tokens 實驗調優，加入 metadata
  - 長文件建立摘要索引（hierarchical indexing）
- **Retrieval 層**
  - 評估 BM25 + Dense 混合方案
  - 使用 Re-ranker（Cohere / Cross-encoder）提升精準度
  - 高頻查詢實施語意快取（Semantic Cache with Redis）
- **Generation 層**
  - Prompt 明確要求只依賴 context，要求引用文件編號
  - 加入「資訊不足時如何回應」的明確指令
- **評估層**
  - 建立 100+ QA pairs Golden Dataset
  - 整合 RAGAs 自動評估，建立 A/B 測試機制

**常見問題排查**

| 問題 | 診斷方向 | 解法 |
|------|---------|------|

### 系統架構總覽

RAG 分為兩個獨立階段：

- **Indexing（離線建索引）**：一次性操作，建好後即可反覆使用
  - 文檔處理 → 嵌入與儲存
- **Querying（線上回答問題）**：每次使用者提問時的即時流程
  - 查詢向量化 → 相似度搜尋 → Prompt 組裝 → LLM 生成

兩個階段共用同一個 Embedding Model

---

### Phase 1：Indexing 資料流（四個核心步驟）

```
01 Document Loader → 02 Text Splitter → 03 Embedding Model → 04 Vector Store
```

| 步驟 | 工具 | 說明 |
|------|------|------|
| Document Loader | PyPDF / TXT | 讀取 TXT / PDF / MD，轉為純文字 |
| Text Splitter | RecursiveCharacterTextSplitter | 切成小段落（Chunks），由 chunk_size 與 chunk_overlap 控制 |
| Embedding Model | text-embedding-3-small | 每個 chunk 轉為 1536 維高維向量 |
| Vector Store | ChromaDB / FAISS | 儲存向量索引 |

> 💡 **關鍵設計決策**：Indexing 是離線一次性操作。只有在知識文件更新時，才需重新執行。

### 為什麼要切 Chunk？

Embedding 模型有 token 長度限制，且過長的文字會稀釋語意向量。

| Chunk 大小 | 優點 | 缺點 |
|-----------|------|------|
| 小（~100 字） | 精準定位特定資訊 | 可能缺乏上下文，語意不完整 |
| **建議（150–300 字）** | 平衡精準度與上下文 | — |
| 大（~500 字） | 上下文完整保留 | 向量語意被稀釋，定位不精準 |

> 📌 **Chunk Overlap 的作用**：相鄰 Chunk 保留重疊區域（建議 chunk_size 的 20%），確保跨段落的關鍵資訊不會因切割而遺失。

### Text Splitter 實作

```python
def load_and_split(filepath: str, chunk_size: int = 200, chunk_overlap: int = 40) -> list[str]:
    with open(filepath, "r", encoding="utf-8") as f:
        text = f.read()
    chunks = []
    start = 0
    while start < len(text):
        end = start + chunk_size
        chunks.append(text[start:end].strip())
        start += chunk_size - chunk_overlap  # 保留 overlap 個字元
    return [c for c in chunks if len(c) > 20]  # 移除空白 chunk
```

### 什麼是 Embedding？

Embedding = 將文字轉換為高維向量，讓電腦能計算「語意距離」。

```
"空氣清淨機濾網多久換一次？" → [0.23, -0.41, 0.87, ...]  ← 1536 維向量
"建議每六個月更換一次濾網"  → [0.21, -0.39, 0.85, ...]  ← 向量接近！
```

### 建立向量索引

```python
import chromadb
from openai import OpenAI

client = OpenAI()

def get_embedding(text: str) -> list[float]:
    response = client.embeddings.create(
        input=text,
        model="text-embedding-3-small"  # 1536 維，便宜且好用
    )
    return response.data[0].embedding

# 建立 ChromaDB
chroma_client = chromadb.Client()  # in-memory（重啟後消失）
# chroma_client = chromadb.PersistentClient(path="./chroma_db")  # 持久化
collection = chroma_client.create_collection(name="my_rag_docs")

def build_index(chunks: list[str]):
    embeddings = [get_embedding(c) for c in chunks]
    collection.add(
        documents=chunks,
        embeddings=embeddings,
        ids=[f"chunk_{i}" for i in range(len(chunks))]
    )
```

---

### 向量搜尋實作

```python
def retrieve(query: str, top_k: int = 3) -> list[str]:
    query_embedding = get_embedding(query)
    results = collection.query(
        query_embeddings=[query_embedding],
        n_results=top_k
    )
    return results["documents"][0]
```

> ⚖ **Top-K 的取捨**：K 越大，召回率高但 Prompt 越長、API 成本越高；K 太小可能漏掉關鍵資訊。實務建議 K = 3 ~ 5，依知識庫密度與問題複雜度調整。

```python
def build_prompt(query: str, contexts: list[str]) -> str:
    context_block = "\n\n---\n\n".join(contexts)
    prompt = f"""你是一位客服助理，只依據以下知識庫回使用者問題。
重要規則：
- 只從「知識庫」中的資訊回答
- 若知識庫中找不到，回覆：「目前我無法回答此問題」
- 不得自行捏造任何資訊

=== 知識庫 ===
{context_block}

=== 使用者問題 ===
{query}

請用中文回答："""
    return prompt
```

> 關鍵原則：Prompt 約束越明確，LLM 幻覺越少。

### 調校技巧

**搜尋結果不準確？**

| 症狀 | 可能原因 | 解法 |
|------|---------|------|
| 搜到不相關的段落 | Chunk 太大，向量被稀釋 | 縮小 chunk_size（試試 100–150 字） |
| 答案跨段落但被切斷 | Chunk 沒有重疊區域 | 增加 chunk_overlap（設為 chunk_size 的 20%） |
| 正確段落沒被撈到 | top_k 設定太小 | 增加 top_k（從 3 調整至 5） |
| 問法與文件用詞差距大 | 語意鴻溝（Lexical Gap） | 加入 HyDE 或 Query Expansion 技術 |

**LLM 自己補充知識庫沒有的資訊？**
→ 在 Prompt 加強約束，明確禁止使用知識庫以外的資訊
